# 轻量 Skill 系统实施计划（Discord 客服 Bot）

## 1. 目标

当前项目已经具备以下能力：
- Discord Bot 在线接收和回复消息
- 按频道使用不同人格 / 语言
- 调用 OpenAI 兼容接口 `/responses`
- 从本地源码目录读取**只读上下文**辅助回答
- 通过 Docker 长期稳定运行

下一阶段的目标，不是把它升级成一个完整的 coding agent 平台，而是把它演进成一个：

**面向客服与问题诊断的轻量 Skill Bot**。

目标效果：
- 对“介绍项目”“功能在哪”“某个 bug 可能出在哪里”这类问题，回答更稳定
- 能根据用户反馈自动读取代码并给出初步诊断
- 能输出结构化、较详细的 issue 草稿
- 保持安全边界：默认只读，不直接改代码，不直接执行危险命令
- Skill 目录与格式尽量兼容标准，方便后续扩展和迁移

---

## 2. 核心判断

### 2.1 为什么需要 Skill 系统

当前简单的“人格 + 关键词检索 + 通用回答”路径，在以下问题上不够稳定：
- “介绍一下这个项目”
- “目录结构是什么”
- “这个功能的实现在哪里”
- “为什么这里会报错”
- “帮我整理成一个 issue”

这些问题的上下文需求并不相同。
如果都走一条统一的 prompt 管线，容易出现：
- 读不到正确上下文
- 读到了上下文但组织方式不对
- 模型给出泛泛回答
- 客服回答和工程诊断混在一起

Skill 系统的作用，就是先判断任务类型，再读取合适的上下文，最后再交给模型生成输出。

### 2.2 为什么不直接引入完整 coding agent

虽然未来的目标包含“读代码诊断并生成 issue”，这确实需要一部分 coding-agent 能力，但当前阶段并不适合直接引入完整 agent runtime。

原因：
- 当前主要场景是客服 + 只读诊断，而不是自动改代码
- 完整 coding agent 往往天然带来 `write`、`edit`、`bash` 等能力
- 权限边界会迅速变复杂
- Discord 客服 Bot 的首要目标仍是稳定在线、可控、低风险

因此，当前更合理的路线是：

**先做只读诊断 agent，再按需演进到受控执行 agent。**

---

## 3. 推荐总体方案

### 3.1 方案结论

采用：

**当前宿主 + 轻量 Skill Router + 标准兼容 Skill 包格式**

不采用：
- 直接把 Pi 作为当前宿主底座
- 直接照搬完整 Codex CLI / 完整 coding agent runtime

### 3.2 设计原则

1. **轻核心**
   - 保留当前 Discord 宿主和 Docker 部署方式
   - 不推翻现有对话链路

2. **能力外置**
   - 将“项目概览”“代码搜索”“诊断”“issue 草稿”等能力做成独立 skill

3. **标准兼容**
   - Skill 目录结构尽量兼容 Codex / Open Agent Skills 思路
   - 使用 `SKILL.md` 作为入口文件

4. **默认只读**
   - 第一阶段仅允许读取目录、搜索文件、读取片段、整理总结
   - 不允许写文件、不允许改代码、不允许 bash 执行

5. **渐进式扩展**
   - 先把“读、查、诊断、整理”做稳
   - 后面再考虑 issue 自动创建、管理员专用执行工具等

---

## 4. Skill 系统的职责边界

### 4.1 Skill 系统负责什么

Skill 系统负责：
- 根据用户问题判断该走哪类能力
- 决定读取哪些类型的上下文
- 决定是否需要目录概览、README、代码片段或结构化模板
- 决定最终回答的组织方式

### 4.2 Skill 系统不负责什么

第一阶段内，Skill 系统不负责：
- 直接修改本地源码
- 直接运行 shell 命令
- 自动提交 GitHub issue
- 自动修 bug
- 引入重型 RAG 或向量数据库

如果未来确实要支持这些能力，应通过“第二阶段：受控执行层”扩展，而不是在第一阶段混进来。

## 5. 目录结构建议

### 5.1 推荐目录

建议新增：

```text
config/
  skills/
    project-overview/
      SKILL.md
      references/
        overview-checklist.md
    code-search/
      SKILL.md
      references/
        search-patterns.md
    bug-diagnosis/
      SKILL.md
      references/
        diagnosis-template.md
    issue-draft/
      SKILL.md
      references/
        issue-template.md
```

### 5.2 为什么放在 `config/skills`

推荐将 skill 维护在 `config/skills`，原因：
- 与当前 `config/prompts` 的组织方式一致
- 便于运维和 Docker 挂载
- 便于以后把 skill 目录整体迁移到仓库外部
- 对项目协作者来说更直观

### 5.3 如何兼容标准位置

如果后续希望更贴近 Codex / Open Agent Skills 的常见扫描方式，建议采用二选一方案：

方案 A：运行时同时扫描以下位置
- `config/skills`
- `.agents/skills`

方案 B：
- 实际维护目录仍为 `config/skills`
- `.agents/skills` 指向 `config/skills`

推荐优先方案 A，因为跨平台和 Docker 环境里比软链接更稳定、可控。

## 6. Skill 包格式建议

### 6.1 最小兼容格式

每个 skill 至少包含：
- `SKILL.md`

可选目录：
- `references/`
- `scripts/`
- `assets/`

### 6.2 `SKILL.md` 的最小内容

建议包含以下字段：

```md
---
name: project-overview
description: 用于回答“项目是做什么的”“目录结构如何”“主要模块有哪些”
version: 0.1.0
triggers:
  - 介绍一下这个项目
  - 这个仓库是做什么的
  - 项目结构
permissions:
  filesystem: read-only
  shell: none
---

# Purpose
说明这个 skill 的目标

# When to Use
说明适用问题类型

# Inputs
说明期望输入，例如用户问题、频道信息、源码根目录

# Read Strategy
说明应优先读取哪些内容，例如 README、.codebase_index.md、package.json、顶层目录结构

# Output Style
说明输出结构，例如：项目定位 / 技术栈 / 目录结构 / 关键模块

# Guardrails
说明边界，例如不要编造、不要假装有写权限
```

### 6.3 为什么采用这种方式

这样做的优点：
- 对人类可读
- 对模型可读
- 后续很容易追加 `scripts/` 或 `references/`
- 能和当前 prompt / 配置体系和平共处
- 未来若迁移到别的 agent 系统，也更容易复用

## 7. 第一阶段建议实现的 Skill

### 7.1 `project-overview`

**用途**
- 回答“介绍一下这个项目”
- 回答“这个仓库是做什么的”
- 回答“有哪些主要模块”
- 回答“目录结构怎么看”

**读取策略**
优先读取：
- `README.md`
- `README.zh-CN.md`
- `.codebase_index.md`（如果存在）
- `package.json` / `Cargo.toml` / `pyproject.toml` 等入口配置
- 顶层目录列表

**输出建议**
- 项目定位
- 技术栈
- 主要功能模块
- 目录结构概览
- 适合谁使用

### 7.2 `code-search`

**用途**
- 回答“某功能在哪”
- 回答“某个模块如何实现”
- 回答“某段逻辑在哪个文件”

**读取策略**
- 关键词提取
- 优先查文件名
- 再查内容命中
- 仅读取少量高相关片段

**输出建议**
- 相关文件路径
- 该文件在整体中的作用
- 与问题最相关的实现点
- 如果证据不足，明确说明不确定

### 7.3 `bug-diagnosis`

**用途**
- 回答“为什么会报错”
- 回答“可能是什么原因导致这个问题”
- 针对用户反馈做初步工程分析

**读取策略**
- 从用户描述抽关键词
- 优先定位实现文件、配置文件、入口文件
- 组合 2~5 个高相关片段
- 结合人格进行简洁诊断

**输出建议**
- 现象复述
- 可能原因（按置信度排序）
- 当前能确认的证据
- 还缺哪些信息
- 下一步建议

### 7.4 `issue-draft`

**用途**
- 将用户反馈整理成较详细的 issue 草稿

**读取策略**
- 结合用户描述
- 结合 `bug-diagnosis` 输出
- 结合项目概览或相关模块信息

**输出模板建议**
- 标题
- 问题描述
- 复现步骤
- 预期结果
- 实际结果
- 初步分析
- 可能影响范围
- 建议补充信息

## 8. Skill 路由方案

### 8.1 路由目标

在收到 Discord 消息后，不再默认总是走统一的“源码片段检索 + 客服回答”路径，而是先做一个轻量分类。

### 8.2 第一阶段建议的路由方式

采用**规则优先的轻量路由**，不要一开始就引入模型分类器。

推荐优先规则：
- 命中“介绍 / 仓库 / 项目结构 / 模块 / 做什么” -> `project-overview`
- 命中“在哪 / 文件 / 实现 / 入口 / 逻辑 / 函数” -> `code-search`
- 命中“报错 / 失败 / 不工作 / 卡住 / bug / why / error / broken” -> `bug-diagnosis`
- 命中“issue / 工单 / 帮我整理 / draft / report” -> `issue-draft`
- 其他 -> `support-answer`（保留现有通用路径）

### 8.3 为什么先用规则路由

优点：
- 成本低
- 结果可预测
- 便于调试
- 适合客服场景

后续如果规则不够，再引入“模型辅助路由”也不迟。

---

## 9. 权限模型设计

### 9.1 第一阶段权限

第一阶段只开放：
- 目录读取
- 文件搜索
- 文本文件读取
- 上下文拼接
- issue 草稿整理

禁止：
- 文件写入
- 文件修改
- shell 执行
- git 操作
- 自动创建 issue

### 9.2 为什么必须坚持只读

因为当前项目的角色是：
- Discord 客服 Bot
- 只读诊断助手

如果过早引入写能力，风险包括：
- 本地源码被误改
- 用户反馈被误解后触发破坏性行为
- Docker 中挂载目录安全边界被削弱
- 调试和审计复杂度上升

### 9.3 第二阶段预留能力

后续如果用户明确需要，可以考虑第二阶段再开放：
- 管理员频道专用 skill
- issue 自动创建
- 只读命令执行（如测试状态收集）
- 明确审批后的 patch 生成

但这些都不应该进入第一阶段。

---

## 10. 与当前人格系统的关系

Skill 不替代人格文件。

二者关系应为：
- **人格文件** 决定“怎么说”
- **Skill** 决定“读什么、怎么组织上下文、输出什么结构”

因此：
- `config/prompts/*.md` 继续保留
- Skill 负责任务路由和上下文选择
- 最终仍然将频道人格和 skill 上下文一起提供给 LLM

这样可以兼顾：
- 中文 / 英文频道差异
- 客服语气一致性
- 工程问题回答更稳定

---

## 11. 与 Docker 部署的关系

Skill 目录建议与当前配置目录一起挂载，使其可在容器外部编辑。

推荐方式：
- 将 `config/skills` 与 `config/prompts` 一起放在宿主机配置目录中
- 容器启动时读取 `/app/config`
- 修改 skill 文件后通过 `docker compose restart` 生效

后续如果想提高可用性，可以再考虑：
- Skill 文件变更热重载
- Prompt 与 skill 的版本标记
- 配置自检日志

但这些不属于第一阶段必要项。

## 12. 实施分阶段计划

### Phase 1：引入 Skill 基础设施

目标：
- 增加 `config/skills`
- 增加 skill loader
- 增加基础的 `SKILL.md` 解析逻辑
- 增加规则路由器
- 保持当前宿主链路不被破坏

交付物：
- `config/skills/project-overview/SKILL.md`
- `config/skills/code-search/SKILL.md`
- `config/skills/bug-diagnosis/SKILL.md`
- `config/skills/issue-draft/SKILL.md`
- skill loader
- skill router

### Phase 2：改造当前上下文注入逻辑

目标：
- 把当前统一的源码上下文构建逻辑改成“按 skill 选择读取策略”
- 为 `project-overview` 增加 README / 目录概览读取
- 为 `code-search` 与 `bug-diagnosis` 增加更稳定的片段提取

交付物：
- 上下文读取策略表
- 项目概览注入器
- 搜索型注入器
- 诊断型注入器

### Phase 3：Issue 草稿结构化输出

目标：
- 让 `issue-draft` 输出统一结构
- 先只生成草稿，不直接提交

交付物：
- issue 草稿模板
- issue 输出格式校验
- 中英双语兼容策略（按频道人格决定）

### Phase 4：观测与安全强化

目标：
- 在日志中记录当前命中 skill
- 记录命中的上下文来源
- 防止 skill 越权读取非预期目录

交付物：
- skill 命中日志
- 只读目录边界检查
- 失败降级策略

## 13. 验证方式

### 13.1 功能验证

至少验证以下问题：

#### `project-overview`
- “介绍一下这个项目”
- “这个仓库是做什么的”
- “主要模块有哪些”

#### `code-search`
- “登录逻辑在哪”
- “导出相关代码在哪”
- “这个功能入口文件是哪个”

#### `bug-diagnosis`
- “为什么这里会报错”
- “某功能点击没反应，可能是什么原因”

#### `issue-draft`
- “帮我整理一个 issue 草稿”
- “根据这个问题写一份 bug report”

### 13.2 安全验证

- 不应写入本地源码目录
- 不应调用 shell
- 不应自动执行 git 操作
- 不应在未命中证据时编造明确结论

### 13.3 部署验证

- 本地直接运行可用
- Docker 运行可用
- 外部挂载 skill / prompt 目录后，重启容器即可生效

## 14. 风险与规避

### 风险 1：Skill 太多导致复杂度反弹

规避：
- 第一阶段只实现 4 个 skill
- 不做可视化工作流编辑器
- 不做通用插件市场

### 风险 2：格式看似标准，实际仍强耦合当前宿主

规避：
- 保持 `SKILL.md` 作为清晰入口
- skill 描述尽量聚焦“任务”和“读取策略”，避免写入大量宿主私有细节

### 风险 3：诊断结果过度自信

规避：
- 在 `bug-diagnosis` 和 `issue-draft` 中强制要求“证据 / 不确定项 / 下一步建议”三段结构

### 风险 4：后续想扩展执行能力时推倒重来

规避：
- 从现在开始就把权限层与 skill 层分开设计
- Skill 元信息中预留 `permissions` 字段

---

## 15. 推荐实施顺序

推荐你实际开发时按以下顺序落地：

1. 先实现 `project-overview`
   - 这是当前最明显的体验短板
2. 再实现 `code-search`
   - 提升“某功能在哪”的回答质量
3. 再实现 `bug-diagnosis`
   - 让客服 Bot 变成“只读诊断助手”
4. 最后实现 `issue-draft`
   - 生成结构化 issue 草稿

理由：
- 按这个顺序，效果是逐步变好且每一步都容易验证
- 同时不会过早引入高风险能力

---

## 16. 最终建议

最终建议只有一句话：

**把当前 Bot 演进成“轻量、只读、标准兼容的 Skill Support Agent”，而不是一次性演进成完整 coding agent。**

这条路线最符合当前项目的实际需求：
- Discord 客服使用场景
- 按频道切换语言 / 人格
- 本地源码只读参考
- Docker 稳定运行
- 后续可逐步扩展到更强的诊断和 issue 生成能力
