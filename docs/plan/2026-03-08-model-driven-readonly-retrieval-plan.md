# 模型驱动只读检索实施计划（含多阶段回复体验）

## 1. 直接结论

当前 “用户问 1 次 -> 机器人答 1 次” 的体验可以保持不变。  
需要同时升级两部分：
- 检索机制：从“静态关键词抽片段”升级为“模型驱动的只读检索循环”
- 交互体验：从“长时间无反馈”升级为“多阶段即时反馈 + 最终答复”

目标是让系统在一次回复流程内完成：
- 先即时确认已收到问题（含表情/短句）
- 先搜证据
- 再读片段
- 最后基于证据作答

而不是依赖我们为每个项目细节打关键词补丁。

---

## 2. 背景与问题

现状：
- 模型无法直接访问本地文件系统
- 宿主先做关键词检索，再把少量片段拼给模型
- 关键词命中不稳定时，模型会“看不到项目里明明存在的事实”

影响：
- 用户输入稍有变化（中英混输、口语表达）就可能漏检
- 回答容易出现“不确定是否支持某功能”的保守结论
- 需要人工持续补词典，维护成本高且不可扩展

---

## 3. 范围与非目标

### 3.1 本期范围（必须完成）

1. 引入多阶段回复体验（用户只发一条问题）
- 可选即时表情反馈（例如：`👀` 表示处理中）
- 先发送一条简短确认语（例如：“收到，我先看一下”）
- 最终再返回完整答案

2. 引入只读工具调用能力
- `list_dir(path)`
- `search_code(query, path, max_results)`（底层用 `rg`）
- `read_file(path, offset, limit)`

3. 引入模型驱动检索循环（内部 1~N 轮）
- 模型先请求工具
- 宿主执行并回填结果
- 达到证据阈值后输出最终答案

4. 保持外部体验单轮
- 用户仍是“发 1 条消息，收 1 条完整答复”

5. 引入“记忆优先，再查源码”的检索顺序
- 先查记忆文档（产品事实、FAQ、发布说明等）
- 若证据不足再进入源码检索

6. 引入证据输出规范
- 回答末尾附“证据来源文件列表（相对路径）”
- 证据不足时明确返回“未找到依据”

7. 加入预算与安全门禁
- 最大工具轮次
- 最大读取文件数
- 最大总字符数
- 仅允许 `source_context.root_dir` 内只读访问

### 3.2 非目标（本期不做）

- 不引入向量数据库
- 不引入文件写入能力
- 不引入命令执行能力（除内部受控 `rg`）
- 不做自动改代码或自动提交 PR
- 不把多阶段回复变成“要求用户多轮输入”的流程

---

## 4. 方案设计

### 4.1 执行流程（对用户单轮，对系统多阶段）

1. 接收用户问题  
2. 立即触发“处理中反馈”
- 尝试添加表情反馈（Discord 支持时启用）
- 发送一条简短确认语（文案按人格/频道配置）
3. 先给模型一段“工具契约说明 + 当前预算”  
4. 模型返回：
- 要么请求工具调用
- 要么直接给最终回答
5. 若请求工具：宿主执行后回填结果给模型
6. 重复第 4-5 步，直到：
- 证据足够，模型返回最终答案
- 或达到预算上限，返回“证据不足”答复
7. 发送最终答案，并更新反馈状态
- 可选将表情更新为 `✅`（成功）或 `⚠️`（证据不足）

### 4.2 检索顺序（Memory First）

统一按以下优先级检索：

1. 记忆文档层（低成本、高确定性）
- `docs/` 下的产品说明、FAQ、发布说明
- `config/prompts/` 中明确的产品事实段（若有）
- 可配置的 `memory` 文档目录

2. 源码证据层（兜底）
- 仅当记忆层证据不足时触发
- 使用只读工具进行目录、搜索、片段读取

### 4.3 工具契约（最小版）

工具返回必须结构化，便于模型稳定消费：

```json
{
  "ok": true,
  "data": "...",
  "truncated": false,
  "meta": {
    "path": "src/...",
    "count": 12
  }
}
```

错误也结构化：

```json
{
  "ok": false,
  "error": "path out of root_dir"
}
```

### 4.4 证据阈值与提前结束

默认阈值（可配置）：
- 至少 1 个命中文件且存在直接相关片段
- 或至少 2 个独立来源相互印证

一旦达阈值，模型立即总结并返回，不再继续检索。

## 5. 配置扩展（建议）

在 `config/discord-bridge.json` 的 `source_context` 下新增：

```json
{
  "retrieval_mode": "model_driven",
  "max_tool_rounds": 3,
  "max_read_files": 6,
  "max_read_chars_total": 12000,
  "search_max_results": 20,
  "memory_first": true,
  "memory_dirs": ["docs", "config/memory"],
  "memory_max_files": 4,
  "memory_max_chars_total": 6000
}
```

兼容策略：
- 未配置时默认走当前 `keyword` 模式，避免一次性破坏线上行为
- 通过开关灰度切换到 `model_driven`

在 `discord` 下新增交互体验配置：

```json
{
  "staged_reply_enabled": true,
  "staged_ack_enabled": true,
  "staged_ack_text_by_channel": {
    "1479760489050275951": "收到，我先看一下。",
    "1479768973095604387": "Got it. Let me check this first."
  },
  "staged_reaction_enabled": true,
  "staged_reaction_processing": "👀",
  "staged_reaction_done": "✅",
  "staged_reaction_fallback": "⚠️"
}
```

## 6. 分阶段任务（Task List）

### Phase 1：多阶段体验最小闭环

1. 在消息处理入口加入“即时反馈”
- 表情反馈（可用则发）
- 简短确认语（按人格/频道可配）
2. 在宿主新增工具执行层（只读）
3. 新增模型驱动检索循环（含轮次预算）
4. 接入统一证据引用格式
5. 增加基础单元测试（路径越界、预算超限、空结果）

### Phase 2：Memory First 与稳定性增强

1. 增加记忆文档读取器与优先检索逻辑
2. 增加工具超时与降级策略
3. 增加日志观测（每次检索轮次、命中文件数、截断次数）
4. 增加“证据不足”模板回复（中英双语可配置）

### Phase 3：灰度与回滚

1. 对指定频道开启 `model_driven`
2. 观察答复正确性与延迟
3. 若异常，单开关回滚 `keyword` 模式

---

## 7. 验收标准（Done Criteria）

1. 用户问“如何连接到 blender”时，回答可引用仓库内已有 Blender Bridge 证据
2. 用户发出问题后，Bot 会先给出即时反馈（表情或短确认）
3. 用户侧无需增加追问轮次，仍保持单问单答
4. 回答包含证据来源文件路径，且路径都在 `root_dir` 内
5. 达到预算上限时，不编造结论，返回证据不足说明
6. 检索顺序满足“先记忆文档，证据不足再查源码”
7. 回归测试通过，不影响现有 persona 与频道路由逻辑

---

## 8. 风险与缓解

风险：
- 检索轮次增加会拉高响应时间
- 模型可能产生无效工具调用
- 过度读取导致 token 成本上升
- 多阶段消息可能带来“重复回复”或噪音感

缓解：
- 强预算限制（轮次/文件/字符）
- 工具调用白名单与参数校验
- 命中阈值后提前结束
- 观测日志持续优化默认值
- 阶段文案保持极短，且可配置关闭

## 10. 回滚方案

若灰度后出现回复质量下降或延迟不可接受：

1. 将 `retrieval_mode` 切回 `keyword`
2. 保留工具层代码但不启用
3. 复盘日志后再逐步调整预算重试

回滚不涉及数据迁移与持久化结构变更，可快速恢复。
